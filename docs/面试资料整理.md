## 二、解决的业务问题

项目针对平台侧推荐与增长场景中的四类核心痛点：

| # | 痛点 | 具体表现 | 本项目解法 |
|---|------|---------|-----------|
| 1 | **高曝光低点击** | 内容获得大量曝光但 CTR 偏低，流量利用率不足 | Dashboard 异常预警 + 漏斗分析前链路诊断 |

| 4 | **复盘协作不统一** | 不同角色对实验结论理解不一致，缺少结构化沉淀机制 | AI 复盘模块：统一输出结论、放量建议、后续动作 |

### 3.2 技术选型理由（面试常问）

**为什么选 Streamlit 而不是 Flask/FastAPI + 前端？**
- Streamlit 与 Python 数据分析链路天然衔接，无需前后端分离
- 开发速度极快，适合 Demo 展示场景，5 天内完成 4 个功能完整的页面
- 内置筛选组件（multiselect、date_input、slider）与图表渲染，无需额外开发
- 便于后续部署到 Streamlit Cloud，零运维成本展示

**为什么用 Plotly graph_objects 而不是 plotly.express？**
- `go`（graph_objects）接口样式完全可控，可统一颜色、字体、tooltip 格式
- 支持双轴图表（CTR/CVR 叠加展示）、自定义漏斗颜色渐变
- `px` 封装过高，在复杂异常标色（如低于均值标红）场景下不够灵活

**为什么选 Claude API 而不是 OpenAI？**
- Claude 在结构化输出遵从度（严格按 Markdown 结构输出）上表现更稳定
- 对中文业务场景的理解更准确，输出内容质量更高
- 设计了完整的降级策略（API 不可用时展示预设示例），保证 Demo 完整性

## 四、AI 技术的具体应用

### 4.1 使用的 AI 模型

- **模型**：`claude-opus-4-5`
- **调用方式**：Anthropic Python SDK，同步调用，timeout=30s
- **调用场景**：AI 异常诊断、AI 实验复盘两个子模块

### 4.2 Prompt Engineering 设计

核心设计原则：**AI 只做结构化总结，不做数值计算**

```
数据计算层（pandas）→ 结构化摘要组装（Python）→ Prompt 构建 → Claude API → 结构化输出渲染
```

**Prompt 设计要点：**

1. **角色设定**：赋予 Claude "资深策略产品经理" 角色，明确其业务背景（推荐系统、增长运营、数据分析）
2. **结构化输入**：将 pandas 计算好的指标摘要、漏斗结果、分群差异以 Markdown 格式注入 Prompt
3. **输出格式约束**：强制要求按固定结构输出（问题摘要/问题位置/原因/影响范围/建议动作/风险提醒）
4. **幻觉控制**：明确禁止"编造数据外的结论"，所有结论必须基于注入的数据
5. **两套独立模板**：异常诊断 Prompt 和实验复盘 Prompt 分别维护，职责清晰

**异常诊断 Prompt 结构：**
```
角色定义 → 当前筛选上下文（日期/场景/人群）→ 核心指标摘要（CTR/CVR/GMV/ROI）
→ 漏斗分析结果（各环节转化率+掉点判断）→ 分群差异摘要（channel/user_type/content_type）
→ 输出格式要求（6个固定模块）
```

### 4.3 降级策略设计

```python
def generate_anomaly_summary(input_data):
    result = _call_claude_api(prompt)
    if result:
        return result, True        # 真实 AI 输出，显示"✨ Claude AI"标签
    return ANOMALY_FALLBACK, False  # 预设输出，显示"📝 示例输出"标签
```

- API Key 未配置 → 直接使用预设输出
- API 调用超时（>30s）→ 使用预设输出
- 网络异常 → 捕获 Exception，使用预设输出
- 页面右侧用 Badge 标注输出来源，保持可读性和可信度

### 4.4 AI 模块风险控制

| 风险 | 控制措施 |
|------|---------|
| 数值计算错误 | 所有数值由 pandas 预计算，AI 仅接收文本摘要，不做计算 |
| 幻觉输出 | Prompt 明确限定"不得引用输入数据外信息" |
| 输出格式不稳定 | 固定 Prompt 模板 + 结构化输出要求，降低随机性 |
| 服务不可用 | 完整降级策略，预设输出保证 Demo 完整性 |
| 敏感信息泄露 | API Key 通过 .env 管理，.gitignore 排除，不进入版本库 |

## 六、产品类面试问题 & 参考回答

### Q1：这个项目解决了什么核心产品问题？

**参考回答：**
这个项目解决的是平台策略团队在**工作协作效率**上的痛点，而不只是数据分析本身。具体来说：

- **问题 1：信息分散**。策略 PM 需要同时看指标看板、漏斗数据、实验结果，工具不统一，分析路径没有引导，效率低。本项目把四个工作节点串成一个完整流程，从 Dashboard 发现问题到 AI 复盘生成结论，有清晰的路径引导。

- **问题 2：实验设计门槛高**。不同人对"这个实验该看什么指标、护栏指标是什么"理解不一，容易漏设或设错。A/B Test 设计助手把这套规则固化成系统逻辑，降低门槛，统一团队认知。

- **问题 3：复盘结论难以对齐**。实验结束后，算法、运营、产品对结论的理解往往不一致。AI 复盘模块输出统一格式的结构化文案，可以直接复制到 Notion 或周报，减少沟通成本。

---

### Q2：你是如何设计这个产品的指标体系的？

**参考回答：**
指标体系分三层设计：

- **主指标**：直接衡量实验目标，按业务目标选择——提升点击效率看 CTR，提升成交转化看 CVR，提升商业结果看 GMV，成本回收看 ROI。

- **辅助指标**：用于理解主指标变化的原因。比如 CTR 提升，同时看 Detail View Rate 判断点击质量是否下降；CVR 提升，同时拆解 Add to Cart Rate、Order Rate、Pay Rate，判断提升来自哪个环节。

- **护栏指标**：防止局部优化损害整体。展示优化类实验必须设 CVR 和 GMV 为护栏，避免 CTR 高但后链路差；补贴类实验必须设 ROI 和 Campaign Cost 为护栏，避免靠成本换结果。

这三层设计来自电商推荐场景的真实实践，核心逻辑是：**主指标验证方向对不对，辅助指标解释为什么，护栏指标保证不出问题**。

---

### Q3：A/B Test 设计助手的产品逻辑是什么？

**参考回答：**
设计助手的核心逻辑是把**人脑里的实验设计规则固化成系统规则**。

具体分三层：

1. **输入收集层**：让用户输入业务目标（提升 CTR/CVR/GMV）、策略类型（展示优化/优惠券/补贴等）、目标场景/人群、实验假设。这些是人工判断的输入。

2. **规则引擎层**：根据业务目标自动匹配主指标，根据策略类型自动匹配辅助指标和护栏指标，根据价格带和成本敏感度推荐观察周期，根据策略类型推荐双组或三组实验。这套规则来自真实实验设计经验的沉淀。

3. **输出层**：生成结构化实验方案（页面卡片 + Markdown 文本），可直接复制到 Notion 或 PRD，减少重复输入和格式整理的时间成本。

这个产品逻辑的价值在于：**把专家经验下沉成系统能力**，让没有丰富实验经验的运营同学也能快速生成标准化的实验草案。

---

### Q4：漏斗分析页是如何判断"前链路问题"还是"后链路问题"的？

**参考回答：**
判断逻辑分两步：

**第一步，绝对阈值判断：**
- CTR < 0.05（5%）→ 前链路问题信号
- CVR < 0.10（10%）→ 后链路问题信号

**第二步，相对阈值判断（与全局均值对比）：**
- CTR 低于全局均值 10% 以上 → 前链路问题
- CVR 低于全局均值 10% 以上，但 CTR 正常 → 后链路问题
- CTR 和 CVR 同时偏低 → 混合型问题
- 两者都正常，但各分群之间差异很大 → 分群差异问题

同时，在漏斗的 6 个环节里，找出相对转化率最低的环节作为"重点掉点"，可以精确到"是加购到下单这一步出了问题"，而不只是笼统说"转化差"。

这种判断逻辑的价值在于：**从感性的"指标不好"变成理性的"哪个环节出了问题，适合做什么策略"**。

### Q6：这个项目与真实平台产品有哪些差距？你如何看待这个差距？

**参考回答：**
差距是真实存在的，我认为主要有三点：

1. **数据是模拟的，不是真实业务数据**。真实数据会有更复杂的噪声、更多异常情况（如节假日、大促峰值），分析逻辑需要更强的鲁棒性。

2. **实验设计是规则驱动的，没有统计显著性检验**。真实的 A/B Test 需要计算样本量、p 值、置信区间，还要处理 CUPED 方差降低等问题。本项目只做了方案结构生成，没有统计引擎。

3. **AI 复盘是辅助总结，不是因果推断**。真实的复盘需要结合业务背景做因果判断，比如结论是不是受外部因素干扰的，这个 Demo 里的 AI 模块做不到。

我认为这个差距是 Demo 的合理边界，项目的价值在于**展示业务理解框架和产品设计思路**，而不是替代真实系统。在面试中，我会主动说明这个边界，而不是刻意回避。

## 七、技术类面试问题 & 参考回答

### Q7：你是如何设计模块化架构的？为什么这么拆？

**参考回答：**
项目按"单一职责"原则拆分成 7 个公共模块：

```
data_loader     → 只负责数据读取和类型处理
metric_calculator → 只负责指标计算（CTR/CVR/GMV/ROI）
filter_utils    → 只负责筛选条件处理
chart_builder   → 只负责图表构建，统一风格
funnel_analyzer → 只负责漏斗计算和问题诊断
abtest_generator → 只负责实验方案生成
ai_summary      → 只负责 AI 输入组装和输出渲染
```

这样拆的原因有三：

1. **防止页面文件膨胀**：4 个页面共用同一套数据和计算逻辑，如果每个页面各自写 CTR 计算，口径容易不一致，维护成本很高。

2. **指标口径集中维护**：所有指标计算都在 `metric_calculator.py` 里，修改一处，全局生效，避免"Dashboard 里的 CVR 和漏斗页里的 CVR 算法不同"的低级错误。

3. **便于扩展**：要加新图表，只改 `chart_builder`；要改 AI 调用模型，只改 `ai_summary`；要加新指标，只改 `metric_calculator`，页面文件不需要动。

### Q8：Streamlit 缓存机制你是如何使用的？

**参考回答：**
使用了 `@st.cache_data` 装饰器，挂在 `data_loader.py` 的 `load_data()` 函数上：

```python
@st.cache_data(ttl=300, show_spinner=False)
def load_data() -> pd.DataFrame:
    df = pd.read_csv(DATA_PATH, parse_dates=["event_date"])
    ...
    return df
```

**缓存的作用：**
- 首次加载 10 万行 CSV 大约需要 1-2 秒，缓存后后续页面跳转不再重新读取
- `ttl=300` 表示缓存 5 分钟后自动失效，适合 Demo 场景（数据不频繁更新）
- `show_spinner=False` 避免在每次页面刷新时展示不必要的加载提示

**注意事项：**
- 缓存的是返回的 DataFrame，每次筛选时对 DataFrame 进行 `.copy()` 操作，避免链式赋值污染缓存对象
- 筛选逻辑不放在缓存函数内，而是在 `filter_utils.apply_common_filters()` 里处理，保持灵活性

### Q9：数据中的漏斗单调性是如何保证的？

**参考回答：**
在数据生成脚本 `generate_data.py` 中，每层数值是用上一层乘以转化率计算的：

```python
click       = impression * ctr_rate
detail_view = click * dvr_rate
add_to_cart = detail_view * acr_rate
order       = add_to_cart * order_rate
pay         = order * pay_rate
```

最后加了一道强制截断保证：

```python
for col_pair in [("click","impression"), ("detail_view","click"), ...]:
    df[col_pair[0]] = np.minimum(df[col_pair[0]], df[col_pair[1]])
```

即使因为随机噪声导致某行数据 click > impression，也会被强制裁剪为不超过上一层，保证漏斗逻辑成立（impression ≥ click ≥ detail_view ≥ add_to_cart ≥ order ≥ pay）。

### Q10：如果数据量增大到 1000 万行，当前架构会有什么性能问题？怎么优化？

**参考回答：**
当前瓶颈在两处：

1. **全量 CSV 读取**：`pd.read_csv()` 每次启动读取整个文件，1000 万行约 1.5-2GB，加载时间会超过 30 秒，超出可接受范围。

2. **pandas 全量聚合**：筛选后对所有数据做 groupby 聚合，行数越多计算越慢，1000 万行下某些分析可能需要 10 秒以上。

**优化方案：**

- **近期方案**：引入 DuckDB（项目已预留 requirements），DuckDB 直接对 Parquet 文件做 SQL 查询，不需要全量加载进内存，10 亿行数据下秒级查询无压力。

- **中期方案**：对 CSV 按日期分区存储（Parquet 格式），筛选时只加载对应分区，结合 `@st.cache_data` 按筛选 key 缓存聚合结果。

- **长期方案**：真实生产场景下接 ClickHouse 或 Hive，分析层 SQL 由数据仓库执行，前端只接收聚合结果，彻底解耦。

### Q11：AI Prompt 模板是如何防止输出不稳定的？

**参考回答：**
用了四个手段：

1. **角色锁定**：明确告知 Claude 它是"资深策略产品经理"，不是通用助手，减少无关内容生成概率

2. **结构化输出约束**：Prompt 里直接写出目标输出结构（`### 问题摘要` `### 问题位置判断` 等），Claude 会严格遵循这个格式

3. **数据边界声明**：在 Prompt 末尾加"所有结论必须基于上方提供的数据，不得引用数据外信息"，减少幻觉

4. **max_tokens 控制**：设置 `max_tokens=1200`，防止输出过长、内容发散

即使如此，仍存在少量不稳定情况（如偶尔漏掉某个模块的标题），这是 LLM 的固有特性。项目的解法是：用预设输出作为降级，保证即使 AI 输出不理想，页面展示也是完整的。

## 八、AI 产品类面试问题 & 参考回答

### Q12：如何评估 AI 模块在产品中的价值？

**参考回答：**
从三个维度评估：

**效率价值**：AI 复盘模块把"看数据、写结论、对齐团队"这三步里的"写结论"自动化。一份人工复盘文档通常需要 30-60 分钟，AI 生成结构化草稿只需要 10 秒，人工只需要审核和补充。效率提升显著。

**质量一致性**：AI 每次输出都按固定结构（问题摘要/原因/影响范围/建议动作），不像人工撰写可能遗漏某项，质量更稳定。

**认知对齐**：统一格式的输出减少了不同角色对同一实验结论的理解差异，产品、算法、运营看到同一份摘要，对齐成本降低。

**评估局限**：AI 在这里的价值是**辅助总结**，不是替代判断。真正的业务判断（这个结论可信吗？是否有外部因素影响？）仍然需要人来做。

---

### Q13：你如何设计 AI 能力的产品边界？

**参考回答：**
这个项目里 AI 能力的边界是：**只做文本生成，不做数值计算和业务判断**。

具体边界：
- ✅ AI 做：基于结构化输入，生成清晰的、结构化的分析摘要文本
- ✅ AI 做：将数据事实转化为可读性强的产品语言
- ❌ AI 不做：计算 CTR、CVR、uplift 等数值（由 pandas 完成）
- ❌ AI 不做：做"应该放量"的最终决策（只输出建议，需人工确认）
- ❌ AI 不做：引入输入数据之外的信息（Prompt 中明确约束）

这个边界设计的核心逻辑是：**AI 最容易出错的地方（数值计算、事实准确性）由确定性代码兜底；AI 最擅长的地方（语言表达、结构整理）才交给 AI**。

这也是目前主流 AI 产品设计的共识：AI as Copilot，而不是 AI as Decision Maker。

### Q14：如果要把这个 Demo 做成真实产品，最优先要补的能力是什么？

**参考回答：**
按优先级排列三点：

**第一：统计显著性检验**。现在 A/B Test 设计助手只能生成方案，不能判断实验结果是否显著。真实产品需要在复盘阶段接入 t 检验、Z 检验，自动给出 p 值、置信区间和"是否显著"的结论。

**第二：实时数据接入**。现在用的是 CSV 静态数据，真实场景需要接 Hive/ClickHouse 的实验数据，支持实验运行中的动态更新，而不是事后分析。

**第三：权限与实验管理**。多人使用时需要区分谁创建了哪个实验、实验的生命周期管理（草稿/运行中/已结束）、实验结论的审核流程。这是从"工具"到"平台"的关键跨越。

### Q16：你在这个项目中遇到的最大挑战是什么？

**挑战**：AI 输出结构稳定性问题。早期 Claude 输出有时会合并两个模块、省略某个标题，导致页面渲染格式混乱。

**解法**：
1. 在 Prompt 里把目标输出结构直接写成"填空题"而非"描述题"——不说"请按照以下结构输出"，而是把结构模板直接放在 Prompt 里，让 Claude 照着填
2. 增加`max_tokens`限制，防止内容发散
3. 设计降级策略，即使 AI 输出不理想，预设输出也能保证页面完整性

**收获**：理解了 Prompt Engineering 的核心不是"让 AI 说对话"，而是"减少 AI 可以说错话的空间"。结构化约束 > 自然语言描述。
